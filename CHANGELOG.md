## [1.0.7] — 2026-05-03

### Added

- **`@pramanasystems/audit-db`** — PostgreSQL audit database client for immutable governance audit trails. Ships as a standalone workspace package (`packages/audit-db`):
  - **4 tables**: `audit_decisions` (full `ExecutionAttestation` as JSONB), `audit_verifications`, `audit_security_events`, `audit_api_access`.
  - **2 views**: `view_decision_timeline` (decisions joined with latest verification status), `view_security_dashboard` (security events aggregated by type/severity).
  - **11 indexes** covering all high-cardinality query patterns.
  - `AuditDb` class — `recordDecision()`, `recordVerification()`, `recordSecurityEvent()`, `recordApiAccess()` are all **fire-and-forget** (never block server responses, never throw).
  - `migrate()` — idempotent schema migration using `CREATE TABLE IF NOT EXISTS` / `CREATE OR REPLACE VIEW`; safe to run on every startup.
  - `getDecisionTimeline()`, `getDecisionById()`, `getVerificationsByExecution()`, `getSecurityDashboard()` — typed read queries.
- **`@pramanasystems/server`** — audit integration:
  - `POST /execute` now calls `auditDb.recordDecision()` fire-and-forget after attestation is produced.
  - `POST /verify` now calls `auditDb.recordVerification()` fire-and-forget after verification completes.
  - `onResponse` Fastify hook records every HTTP request to `audit_api_access` and emits a `severity: medium` `auth_failure` security event for every 401.
  - **4 new audit query routes** (registered only when `AUDIT_DATABASE_URL` is set): `GET /audit/decisions`, `GET /audit/decisions/:executionId`, `GET /audit/security`, `GET /audit/verifications/:executionId`.
  - Route logic extracted to `src/routes/execute.ts`, `src/routes/verify.ts`, `src/routes/audit.ts`; middleware in `src/middleware/audit.ts`.
- **`docker-compose.yml`** — `postgres:16-alpine` service (`pramana_audit` database) with healthcheck; server `depends_on` postgres `service_healthy`; `AUDIT_DATABASE_URL` wired automatically.
- **`.env.example`** — `POSTGRES_PASSWORD` and `AUDIT_DATABASE_URL` documented.
- **`packages/audit-db/tests/audit-db.test.ts`** — 18 vitest unit tests with mocked `pg.Pool` covering all fire-and-forget methods, read queries, migration transaction handling, and error swallowing.

## [1.0.6] — 2026-05-03

### Added

- **`@pramanasystems/dashboard`** — React 18 + Vite + Tailwind CSS single-page application for interacting with the governance runtime via `@pramanasystems/sdk-client`. Four views:
  - **Health** — live `GET /health` polling every 10 seconds with status badge and last-checked timestamp.
  - **Execute** — `POST /execute` form with syntax-highlighted `ExecutionAttestation` response and decision colour badge.
  - **Verify** — paste or pipe an `ExecutionAttestation` JSON; shows per-check pass/fail indicators (`signature_verified`, `runtime_verified`, `schema_compatible`) and an overall VALID/INVALID banner.
  - **Runtime** — Coming soon placeholders for `GET /runtime/manifest` and `GET /runtime/capabilities`.
  - Auth badge shows **Authenticated** / **Dev Mode** based on whether an API key is configured.
  - Base URL and API key configurable at runtime via settings panel; persisted in `localStorage`.
  - `private: true` — not published to npm.
- **`npm run dashboard`** root workspace script — starts the Vite dev server for `packages/dashboard`.
- **JSDoc comments** on all exported functions, classes, and types across all eight packages (`bundle`, `crypto`, `governance`, `execution`, `verifier`, `core`, `server`, `sdk-client`).
- **Documentation rewrite** — all six `docs/` files rewritten to production quality: `getting-started.md`, `verification.md`, `architecture/ARCHITECTURE.md`, `trust/TRUST_MODEL.md`, `trust/THREAT_MODEL.md`, `verification/RELEASE_VERIFICATION.md`.
- **Package READMEs** — individual READMEs for all nine packages (`bundle`, `crypto`, `governance`, `execution`, `verifier`, `core`, `server`, `sdk-client`, `verifier-cli`).

## [1.0.5] — 2025-05-02

### Added

- **`@pramanasystems/server`** — Fastify v5 REST API server wrapping the deterministic execution runtime. Exposes `POST /execute`, `POST /verify`, `GET /health`, and four stubbed extensibility endpoints. Bearer token auth via `PRAMANA_API_KEY` (skipped in dev mode). Supports `PRAMANA_PRIVATE_KEY`/`PRAMANA_PUBLIC_KEY` env vars or falls back to an ephemeral Ed25519 keypair.
- **`@pramanasystems/sdk-client`** — Zero-dependency TypeScript fetch client for the server API. Types generated from `openapi.json` via `openapi-typescript`. Exports `PramanaClient` (with `health()`, `execute()`, `verify()`), `PramanaApiError`, and all request/response types.
- **`openapi.json`** — Full OpenAPI 3.0.3 specification covering all 7 server routes, generated by `@fastify/swagger`.
- **AWS KMS integration tests** at `tests/integration/aws-kms/` — seven tests covering sign/verify roundtrip, cross-payload isolation, tamper detection, full governance execution with `AwsKmsSigner`, attestation verification, and replay protection. Skipped automatically when `AWS_KMS_KEY_ID` is absent.
- `vitest.config.ts` — `hookTimeout` and `testTimeout` raised to 60 s to accommodate real AWS API calls in CI.

## [1.0.4] — 2025-04-18

### Added

- **SDK-friendly APIs** on `@pramanasystems/execution` and `@pramanasystems/core`:
  - `executeSimple()` — minimal one-call execution returning an `ExecutionAttestation`
  - `verifyPayloadSignature()` — standalone signature verification helper
  - `definePolicy()` — typed policy definition builder
- **Deterministic governance validator architecture** in `@pramanasystems/core` — canonical form validation, invariant checks, and validator configuration types.
- Hardened deterministic verification and reproducibility guarantees across the execution pipeline.
- Import boundary enforcement in the check pipeline.

### Changed

- `@pramanasystems/core` now re-exports `definePolicy` from `@pramanasystems/governance`.
- Strengthened `hashRuntime()` determinism across Node.js versions.

## [1.0.3] — 2025-03-01

### Added

- `@pramanasystems/verifier` published as a standalone package — portable, independently executable attestation verification with no runtime trust dependency.
- `verifyAttestation()` — three-check verification: signature, runtime hash, schema compatibility.
- `verifyBundle()` and `verifyRuntime()` — bundle and runtime manifest verification.
- `@pramanasystems/verifier-cli` — command-line tool for offline attestation, release, and runtime verification.
- `MemoryReplayStore` — in-memory replay protection store.
- `LocalSigner` / `LocalVerifier` — Ed25519 signing and verification using Node.js `crypto`.
- `getRuntimeManifest()` / `signRuntimeManifest()` / `verifyRuntimeManifest()` — runtime provenance chain.

### Changed

- `@pramanasystems/execution` version bumped to align with verifier release.
- Canonical serialization hardened — `canonicalize()` now produces RFC 8785 JSON Canonicalization Scheme output.

## [1.0.0] — 2025-01-15

### Added

Initial public release of the PramanaSystems deterministic governance ecosystem:

- **`@pramanasystems/bundle`** — deterministic artifact canonicalization, hashing, manifest generation, and bundle read/write.
- **`@pramanasystems/crypto`** — Ed25519 key generation, persistence, signing, and verification primitives.
- **`@pramanasystems/governance`** — policy creation, upgrade, validation, bundle generation, and execution requirements.
- **`@pramanasystems/execution`** — full deterministic execution runtime: execution tokens, token signing/verification, execution results, attestation building, replay store interface.
- **`@pramanasystems/core`** — portable SDK re-exporting governance lifecycle, execution, and verification in a single install.
- Deterministic execution invariants: token issuance, canonical signing, replay protection, fail-closed semantics.
- `docs/` — architecture, governance, trust model, threat model, verification, and release verification documentation.
- `examples/` — quickstart and decision-flow reference implementations.
